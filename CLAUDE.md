# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Vue 3 + TypeScript date range picker distributed as a library. `vite build` produces ES + CJS bundles (`dist/date-range-picker.{js,cjs}`) plus a single `dist/style.css` and type declarations generated by `vite-plugin-dts`. Vue is a peer dependency and is externalised from the bundle. Public entry point: `src/index.ts`. `src/App.vue` + `src/main.ts` exist only as a local demo (excluded from the built types).

Package manager: Yarn (see `.yarnrc.yml`). Import alias `@/` → `src/` is defined in `vite.config.ts` and `tsconfig.app.json` — use it in new code.

## Commands

- `yarn dev` — run the Vite demo app (`src/App.vue`).
- `yarn build` — build the library (emits to `dist/`, including `.d.ts`).
- `yarn typecheck` — `vue-tsc --noEmit` across the project references.
- `yarn test` — run all Vitest specs once (happy-dom environment).
- `yarn test:watch` — Vitest in watch mode.
- Run a single spec: `yarn test src/components/atoms/CalendarDay/CalendarDay.spec.ts` (pattern match also works, e.g. `yarn test CalendarDay`).

Tests use `@vue/test-utils` and are co-located with the file under test as `*.spec.ts` (the Vitest `include` glob is `src/**/*.spec.ts` — `.test.ts` files are ignored).

## Development workflow

Applies to any **non-trivial task** (anything beyond a typo or a one-line fix on an already-checked-out branch). The goal is that multiple Claude sessions can work in parallel in isolated worktrees without colliding.

1. **Always start from an up-to-date `main`.** Before creating a new branch or worktree:

   ```
   git fetch origin
   ```

   Base the new branch on `origin/main`, not the current `HEAD`.

2. **One task = one branch = one PR.** Branch name: `<type>/<kebab-name>` where `<type>` is one of `feat`, `fix`, `chore`, `refactor` (inferred from existing `git log`). Example: `feat/slot-customization`.

3. **Work in a dedicated git worktree** so parallel sessions don't step on each other:

   ```
   git worktree add ../date-range-picker-<short-name> -b <type>/<short-name> origin/main
   ```

   After the PR is merged, clean up:

   ```
   git worktree remove ../date-range-picker-<short-name>
   ```

   Never switch branches inside an existing worktree to "save time" — open a second worktree instead.

4. **Gate before opening the PR.** Both must pass locally:

   ```
   yarn typecheck && yarn test
   ```

   Never bypass with `--no-verify`.

5. **Conventional Commits with optional scope**, matching the existing `git log` style: `feat(calendar-grid): …`, `fix(drag): …`, `chore: …`. Lowercase subject, no trailing period.

6. **PR targets `main`.** Title = the primary commit subject. Body has a `## Summary` section and a `## Test plan` checklist.

7. **Scope discipline: one PR = one concern.** If an unrelated issue surfaces mid-task, note it and move on — open a separate PR rather than bundling.

8. **Delegated agents that do real work** (not just research) are spawned with `isolation: "worktree"` so they operate on their own copy of the repo, independent of the session's worktree.

## Architecture

### State lives in a composable, components are dumb

All picker logic is in `src/composables/useDateRangePicker/useDateRangePicker.ts`. The organism `DateRangePicker.vue` instantiates the composable once, wires its v-model `start`/`end` into `committedStart`/`committedEnd`, and fans the returned refs/handlers out to child components via props and events. Child components (molecules, atoms) hold no picker state — when adding a feature, extend the composable and expose a new ref/handler rather than adding state to a component.

### Three-mode state machine

`PickerState = "idle" | "selecting" | "selected"` drives everything:

- **idle** — only committed dates render as a range.
- **selecting** — an `anchor` date has been clicked; the second click transitions to `selected`. Navigation in this mode pins whichever calendar currently shows the anchor (see `navigatePrev`/`navigateNext`).
- **selected** — tentative range shown, with `commit`/`reset` exposed by `ActionBar`. Drag-to-adjust (endpoint drag or whole-range drag) is only enabled here; `previewRange()` overlays the in-flight range on top of `tentativeStart`/`tentativeEnd` without mutating them until `commitDrag()`.

`reset()` restores `committedStart`/`committedEnd` from the snapshot taken when the range first entered `selected`, so the v-model reverts to what the parent had committed before this selection round — not to empty.

### Two-calendar invariants

The picker always shows two months, `leftMonth` and `rightMonth`. `applyYearMonth` (used by the month/year pickers) enforces `left < right` by bumping the other side when needed. `navigatePrev`/`navigateNext` usually move both months in lock-step, except during `selecting` where the calendar showing the anchor is pinned. `viewSelection()` jumps both calendars to frame the tentative range (same month → left=that month, right=next; different months → left=start month, right=end month).

### Drag-to-page during drag

When dragging an endpoint, `dragSourceSide` records which calendar the dragged boundary started on. Hovering the prev/next arrows while dragging triggers `pageSourcePrev`/`pageSourceNext`, which advance only the source side (and are blocked from crossing the other side). `CalendarNavigation.vue` owns the hover timers (`AUTO_PAGE_INITIAL_DELAY_MS`, `AUTO_PAGE_REPEAT_DELAY_MS`) and emits `auto-page-prev`/`auto-page-next`.

### Month grid conventions

`generateMonthGrid(year, month)` in `calendar-utils.ts` always returns a **6×7 grid starting on Monday**, padded with adjacent-month days. `YearMonth.month` is **1-indexed** (1 = January) — convert to/from `Date` via `yearMonthFromDate` / `new Date(year, month - 1, ...)`. Compare months with `compareYearMonth`, never by subtracting fields directly.

`DayInfo` is the per-cell view model produced by `computeDayInfo`. Atom components (`CalendarDay`, etc.) should consume `DayInfo` rather than re-deriving flags like `isInRange` from raw dates.

### Atomic-design layout

```
src/components/
  atoms/      — Button, CalendarDay, CalendarMonthCell, CalendarMonthHeader,
                CalendarWeekdayRow, CalendarYearCell, NavArrow
  molecules/  — ActionBar, CalendarMonth, CalendarNavigation
  organisms/  — DateRangePicker (the only public component)
```

Each component folder has `ComponentName.vue`, `types.ts` (props/emits types), `index.ts` (barrel), and `ComponentName.spec.ts`. `src/index.ts` re-exports only what is public (`DateRangePicker`, `Button`, and the `PickerState`/`YearMonth`/`DayInfo`/`MonthGrid` types from the composable).

### Localisation

UI strings default to French (`locale: "fr-FR"`, action-bar buttons "Valider" / "Reset" / "Voir la sélection" are hard-coded in `ActionBar.vue`). Month and weekday names go through `toLocaleString` in `calendar-utils.ts` and respect the `locale` prop propagated from `DateRangePicker`. If adding user-facing strings, match the French/i18n split already in place (format via `locale` if it's a date string; otherwise it's currently hard-coded French).

### Styling

Components use scoped styles with CSS custom properties prefixed `--drp-*` that fall back to generic `--border`, `--bg`, `--sans`, etc. Keep the `--drp-*` prefix when adding new theme hooks so consumers can override them without clashing with their own design tokens. All CSS ends up in a single `dist/style.css` (`cssCodeSplit: false`).
